## 2. 产品概述

本系统定位为面向仓储/物流分拣作业编排的业务工作流平台，用于承接从订单进入仓内作业，到组波、扫描、RCS 调度、分拣、集包、打印、贴码的完整分拣链路。

系统向上对接 `OMS`、`ERP/TMS`、`WMS` 等业务系统，向下对接 `RCS`、分拣设备控制系统、打印系统、贴码系统，并统一承接人工工位、PDA、机械臂等执行主体。系统的核心价值不是替代上下游专业系统，而是在它们之间建立一层可配置、可编排、可追溯的业务流程平台，使客户差异更多通过配置和受控扩展承接，而非持续依赖项目化硬编码开发。

## 3. 业务背景与立项动因

### 3.1 上下游背景

- 上游系统：`OMS` 订单系统、`ERP/TMS`、`WMS`
- 下游系统：`RCS` 机器人调度、分拣设备控制、打印系统、贴码系统

### 3.2 主流程背景

标准主流程如下：

`接受订单 -> 根据订单组波次 -> 扫描进分拣框 -> RCS -> 分拣至料口 -> 集包 -> 打印 -> 贴码`

其中，“波次”指一定时间段内多个订单组成的业务处理批次。不同客户在组波规则、指定料口、优先级、打印方式、贴码方式、异常处理方式上差异显著。

### 3.3 核心业务问题

- 客户分拣规则高定制化，难以通过固定系统流程覆盖
- 主流程各环节高度差异化，组波、打印、贴码常由客户已有系统承担
- 现场执行主体多样，人工、PDA、机械臂、外部设备混合存在
- 项目落地过程中接口联调、异常排查和流程调整成本高
- 新客户上线时复用能力弱，容易演变为一次一套的定制交付

### 3.4 立项目标

建设一套平台化工作流系统，支撑多仓、多客户、多套组波/打印/贴码规则并行落地，通过可视化流程编排、规则中心和插件扩展，实现高灵活定制能力，并提升项目复用率和交付效率。

## 4. 产品目标与非目标

### 4.1 产品目标

- 提供可视化流程编排能力，支持主流程、条件分支、异常分支、人工节点、设备节点和回调等待节点配置
- 提供平台化规则承接能力，覆盖组波、料口分配、任务路由、打印、贴码、异常处理等关键环节
- 支持多仓、多客户、多业务场景的流程模板和规则隔离
- 统一承接人工、PDA、机械臂和外部系统执行任务
- 通过标准连接器和集成网关降低项目接口改造成本
- 提供生产可运营能力，包括监控、告警、审计和问题追踪

### 4.2 首版优先级

首版产品的第一优先级为 `高灵活定制能力`，在此基础上兼顾项目交付效率和生产可运行性。

### 4.3 非目标

- 不替代上游 `OMS/WMS/ERP/TMS` 的主数据、交易、库存或运输管理能力
- 不替代下游 `RCS`、设备控制系统本身的专业调度和控制能力
- 不在首版中建设复杂经营分析、算法优化平台或资源仿真平台
- 不要求客户必须替换其已有组波、打印、贴码系统，允许通过集成纳入整体流程

## 5. 用户角色与典型场景

### 5.1 核心角色

- 产品经理/解决方案经理：定义行业模板、客户差异能力边界和产品演进方向
- 实施/交付工程师：完成流程配置、规则上线、接口联调和现场调试
- 仓内运营主管：关注波次状态、任务积压、异常处置和仓内作业节奏
- 一线作业人员：通过工位终端或 PDA 执行扫描、集包、打印确认、贴码确认等任务
- 集成/运维人员：负责上下游接口、设备调用、告警处理、日志追踪和发布运维

### 5.2 典型业务场景

- 多客户差异化流程快速落地：不同客户使用不同组波规则、料口策略、打印贴码链路
- 流程动态调整：在标准主流程中插入人工复核、等待回调、切换备用料口或跳过指定环节
- 多执行主体统一承接：相同业务节点可由人工、PDA、机械臂或外部系统完成
- 生产异常闭环：设备失败、回调丢失、打印异常、任务超时后可重试、补偿或人工接管

### 5.3 成功标准

- 新客户项目可基于标准模板快速复制流程骨架
- 客户差异主要通过规则配置和少量插件扩展承接，而非大规模修改主代码
- 订单到贴码的关键节点全链路可视、可查、可追责
- 现场接口联调、流程调试和异常定位拥有统一工具与模型
- 多仓、多客户环境下流程、规则、资源和数据隔离清晰

## 6. 产品范围与 MVP 边界

### 6.1 首版覆盖范围

首版覆盖从订单接入到贴码完成的主业务链路，并支持以下平台能力：

- 可视化流程编排
- 规则中心
- 波次管理
- 作业任务中心
- 上下游集成网关
- 监控告警与审计

### 6.2 首版必达业务闭环

首版需打通以下标准闭环：

`订单接入 -> 组波 -> 扫描入框 -> RCS/设备调度 -> 分拣至料口 -> 集包 -> 打印 -> 贴码 -> 状态回传/完结`

并支持与之对应的异常处理、人工干预和回溯追踪。

### 6.3 后续版本增强项

- 复杂运营分析与经营报表
- 高级容量评估与资源仿真
- 更丰富的行业模板资产库
- 更多标准连接器和插件市场能力
- 跨仓协同与集团级统一运营能力

## 7. 系统总体方案

系统采用“`流程编排内核 + 规则决策中心 + 作业任务中心 + 集成网关 + 运营可观测能力`”的总体方案。

系统不将客户流程固化为代码，而是将业务链路抽象为可编排节点，将客户差异抽象为规则，将高度定制部分收敛到受控插件扩展点，再通过统一任务模型连接人工与自动化执行主体。

### 7.1 总体架构分层

- 接入层：承接上游订单、库存、运输、履约上下文
- 编排层：定义流程模板、节点关系、条件分支、异常流转
- 决策层：执行组波、路由、料口、打印、贴码等业务规则
- 执行层：下发人工任务、PDA 任务、设备调用和外部系统调用
- 集成层：统一管理 API、消息、回调、重试、幂等和协议适配
- 运营层：提供监控、告警、日志、审计、发布和配置治理

### 7.2 核心产品原则

- 配置优先：能配置的不写插件
- 插件兜底：复杂逻辑通过受控扩展点承接
- 模板复用：标准模板派生客户/仓级实例
- 统一任务：不预设执行主体，统一抽象任务模型
- 可追溯：每个订单、波次、任务、接口调用都可被追踪

## 8. 核心模块设计

### 8.1 可视化流程编排器

用于配置主流程、节点顺序、条件路由、异常分支、回调等待和人工确认流程。

支持的节点类型包括：

- 系统处理节点
- 规则决策节点
- 人工作业节点
- 外部系统调用节点
- 设备执行节点
- 等待回调节点
- 异常补偿节点

核心能力包括：

- 流程模板设计、保存、发布、回滚
- 模板按租户/客户/仓派生
- 条件分支与异常流转配置
- 节点参数配置与资源绑定
- 版本管理与灰度发布

### 8.2 规则中心

用于承接客户差异逻辑，首版覆盖：

- 组波规则
- 指定料口/料口分配规则
- 任务路由规则
- 打印规则
- 贴码规则
- 异常分流与补偿规则

规则模型支持：

- 条件表达式
- 优先级与命中顺序
- 默认兜底策略
- 参数化配置
- 规则调试与版本管理

### 8.3 波次管理

负责将订单按业务规则聚合为波次，并管理波次生命周期。

支持能力包括：

- 定时组波
- 阈值触发组波
- 人工触发组波
- 外部系统组波结果回填
- 波次优先级、容量、时间窗、指定料口管理
- 波次状态追踪和人工干预

### 8.4 作业任务中心

负责将流程节点实例化为统一任务，并派发给不同执行主体。

核心能力包括：

- 统一任务模型与状态机
- 人工工位/PDA/机械臂/外部系统任务分发
- 任务签收、执行、回执、超时、失败、重试
- 任务与流程实例、波次、订单的关联追踪

### 8.5 上下游集成网关

负责对接 `OMS/ERP/TMS/WMS` 与 `RCS/设备/打印/贴码` 等外部系统。

核心能力包括：

- API 接入与标准协议封装
- 消息订阅与事件分发
- 回调接收与状态回填
- 幂等控制、超时控制、重试机制
- 错误码规范和调用追踪
- 连接器与协议适配器管理

### 8.6 监控告警与审计

用于支撑生产可运营能力。

核心能力包括：

- 流程实例监控
- 节点耗时分析
- 波次积压与任务阻塞监控
- 接口异常告警
- 设备失败告警
- 人工接管记录
- 全链路审计追踪

## 9. 核心业务流程设计

### 9.1 标准主流程

1. 上游系统推送订单及业务上下文
2. 系统完成订单标准化、幂等校验和流程模板匹配
3. 根据规则生成波次，或调用外部组波系统并回填结果
4. 下发扫描入框任务，记录订单、分拣框、波次、工位关系
5. 调用 `RCS` 与分拣设备执行调度与分拣动作
6. 将货物送达指定料口或规则命中的动态料口
7. 生成集包、打印、贴码任务，由统一任务模型承接
8. 接收执行回执，驱动流程闭环并同步状态给上下游系统

### 9.2 可编排点

首版需支持以下可编排能力：

- 在主流程中插入人工复核节点
- 根据条件切换不同组波策略
- 根据仓内状态切换料口或执行路径
- 将打印、贴码配置为内部执行或外部系统执行
- 根据客户要求跳过、替换或延后部分节点

### 9.3 异常处理能力

首版需支持以下异常类型：

- 订单数据缺失或格式异常
- 组波失败或组波结果不满足约束
- 扫描异常、条码异常、框号不匹配
- `RCS` 调度超时或失败
- 设备执行失败、料口不可用
- 打印失败、贴码失败
- 外部系统回调缺失或状态冲突
- 任务超时无人处理

异常处理策略包括：

- 自动重试
- 挂起等待
- 切换备用路径
- 转人工处理
- 人工强制继续或终止
- 产生告警并保留审计记录

## 10. 数据对象与配置模型

### 10.1 核心业务对象

- 订单
- 订单行/包裹
- 波次
- 分拣框
- 料口
- 工位
- 设备资源
- 流程模板
- 流程实例
- 节点实例
- 作业任务
- 规则集
- 插件定义
- 异常事件
- 审计记录

### 10.2 配置组织层级

系统按以下层级组织配置：

`租户 -> 客户 -> 仓 -> 业务场景 -> 流程模板/规则集/资源配置`

该模型需要满足：

- 集团级模板可复用
- 客户级差异可隔离
- 仓级资源可独立映射
- 配置可版本化、可发布、可回滚

### 10.3 规则模型分类

- 决策规则：决定走哪条流程、命中哪个料口或哪个执行路径
- 计算规则：生成波次、优先级、打印参数、贴码参数等结果
- 约束规则：限制容量、时间窗、客户特殊限制和资源可用性

## 11. 插件扩展机制

### 11.1 设计目标

在保持平台治理能力的前提下，承接配置无法表达的复杂客户逻辑。

### 11.2 首版扩展点

- 组波计算扩展
- 料口分配扩展
- 打印参数加工扩展
- 贴码数据转换扩展
- 异常补偿策略扩展

### 11.3 治理原则

- 插件仅可运行在平台预定义扩展点
- 插件输入输出必须遵循统一对象契约
- 插件执行受超时、权限和日志控制
- 插件版本需支持发布、回滚和审计
- 插件是平台能力补充，不是替代标准流程的自由编码入口

## 12. 系统接口边界

### 12.1 上游接口边界

系统接收并标准化以下信息：

- 订单信息
- 履约信息
- 库存或仓内上下文
- 运输属性
- 客户/仓/业务标识

系统不承担上游主数据和交易闭环管理，仅负责将外部数据转化为可编排的业务上下文。

### 12.2 下游接口边界

系统向下游输出以下能力：

- 业务指令下发
- 任务触发
- 状态回写
- 异常处理编排

系统不替代 `RCS` 或设备控制逻辑，只负责业务层编排和状态治理。

### 12.3 支持的集成模式

- `HTTP/HTTPS API`
- 消息队列/事件订阅
- 回调通知
- 文件/表接口兼容模式

所有集成模式统一纳入接入网关治理，包括认证鉴权、幂等、超时、重试、错误码和调用日志。

## 13. 部署架构与运行模式

### 13.1 部署形态

系统需要兼容两类部署模式：

- 客户现场私有化部署
- 集团统一云平台 + 边缘节点部署

### 13.2 云边协同要求

- 中心平台负责流程模板、规则版本、配置治理和统一监控
- 边缘节点负责低时延设备交互和现场任务执行
- 中心与边缘之间支持模板、规则、资源映射和审计信息同步
- 边缘节点在中心短暂不可用时可维持核心作业连续性

## 14. 非功能需求

### 14.1 可用性

- 核心编排、规则执行、任务分发和集成调用链路需具备故障恢复能力
- 单点故障不应导致整仓作业链路完全不可用

### 14.2 可追踪性

- 订单、波次、流程实例、节点实例、任务实例、接口调用需全链路关联
- 任一异常事件都应可追溯到来源、处理动作和处理结果

### 14.3 安全与隔离

- 多客户、多仓数据与配置严格隔离
- 插件执行权限受控
- 关键操作具备审计记录

### 14.4 可运维性

- 提供日志、指标、告警、审计与配置变更记录
- 支持按客户、仓、流程模板、波次、订单维度定位问题

### 14.5 扩展性

- 新仓、新客户、新流程模板和新连接器接入不应引起底层架构重构
- 流程、规则、资源和插件能力可按模块横向扩展

### 14.6 性能要求口径

本 PRD 首版定义能力边界与平台要求，不在此文档中约束具体吞吐数值与 SLA 数值。具体容量指标由样板仓业务规模、设备数量和订单峰值在详细技术方案阶段量化。

## 15. 版本规划

### 15.1 Phase 1：标准主流程产品化

- 打通标准主流程
- 建立流程模板、规则模板和统一任务模型
- 完成关键上下游标准接入

### 15.2 Phase 2：差异化承接能力增强

- 补齐插件扩展点
- 提供模板派生、规则调试和灰度发布能力
- 丰富异常补偿和现场调试工具

### 15.3 Phase 3：平台运营化

- 沉淀模板资产库和连接器资产
- 加强多仓复制能力
- 建设更完善的运营分析和集团级视图

## 16. 实施建议

- 以一个具有代表性的客户或仓作为样板场景先行验证
- 样板场景应同时包含标准流程和至少 1 到 2 个明显定制点
- 研发与交付双线推进：一条建设平台通用能力，一条验证现场落地闭环
- 产品上线策略以“标准模板先跑通，再逐步放开复杂扩展”为主

## 17. 立项结论

本项目适合按平台化产品立项，而非单项目系统开发立项。其核心价值在于建立仓储/物流分拣业务的统一编排底座，将高定制业务场景沉淀为可复用的流程、规则、连接器和扩展能力，从而支撑多仓、多客户、多规则场景下的快速复制与持续演进。

首版产品建议围绕“高灵活定制能力”展开建设，以可视化流程编排器为核心，以规则中心和插件扩展为差异化承接手段，以作业任务中心和集成网关为落地抓手，以监控告警和审计能力保障现场可运营性。
