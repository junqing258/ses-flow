## 2. 一句话介绍项目

本项目是一个面向仓储/物流分拣场景的作业编排平台，在 `OMS/WMS/ERP/TMS`、`RCS/设备/打印/贴码系统` 以及人工工位、PDA、机械臂之间，提供一层可视化、可配置、可追溯的流程执行能力。

项目目标不是替代上下游专业系统，而是把客户差异化流程从“项目代码硬编码”逐步收敛为“流程配置 + 规则判断 + 受控扩展”。

## 3. 当前项目的主要功能

### 3.1 面向用户可见的能力

- 提供工作流列表、创建、编辑、发布和运行入口。
- 前端编辑器支持 `Edit / Run / AI` 三种模式切换。
- 支持通过画布拖拽和节点面板配置流程结构、连线和节点参数。
- 支持运行流程、查看运行状态，并在页面中查看 timeline。
- 支持 AI 编辑会话：前端可创建会话、展示 `runner_base_url` 与 `session_id`、接收最新草稿预览。

### 3.2 当前已落地的运行时能力

- 支持工作流注册、查询、列表、目录刷新和运行历史查询。
- 支持运行实例启动、等待恢复、恢复执行、终止执行。
- 支持运行摘要、节点 timeline、SSE 事件流订阅。
- 支持子流程执行、任务节点等待、Wait 节点等待和事件恢复。
- 支持 workflow、run、snapshot、edit session 的持久化。
- 支持在 backend 进程内直接装配 runner 内核，无需额外 runner 服务进程。

### 3.3 当前已经支持的主要节点

Runner 已实现的节点类型包括：

- `start`
- `end`
- `webhook_trigger`
- `respond`
- `fetch`
- `set_state`
- `if_else`
- `switch`
- `code`
- `shell`
- `wait`
- `task`
- `sub_workflow`

其中，前端调色板当前默认开放的主要节点包括：

- `Start / End`
- `Webhook Trigger / Respond`
- `Fetch`
- `Switch`
- `Sub-Workflow`
- `Shell / Code`
- `Wait`
- `Task`

需要明确说明的边界：

- `set_state`、`if_else` 已有运行时支持，但前端未作为正式节点默认开放。
- `Webhook Trigger` 当前更偏流程定义元数据，不是“配置完即自动生成独立 HTTP 路由”。
- AI 模式当前是“外部 Agent + 编辑会话 + Web 展示”的协作模式，还不是页面内原生聊天式编辑。

## 4. 这个项目现在解决了什么问题

从业务上看，项目已经把“流程定义”和“流程执行”从传统项目代码里抽离出来，初步建立了一个可复用的流程执行底座。

从技术上看，项目已经不只是静态画图工具，而是形成了一个最小闭环：

`流程定义 -> 注册/发布 -> 运行 -> timeline -> waiting -> resume -> 终止/完成`

从平台化演进上看，项目已经具备以下几个关键起点：

- 有统一 DSL 和节点协议，而不是每个项目一套逻辑。
- 有等待恢复模型，而不是只能顺序执行。
- 有运行快照和 timeline，而不是“出了问题只能翻日志”。
- 有子流程、任务和 AI 会话能力，说明后续具备扩展空间。

## 5. 跨部门痛点


### 5.1 产品 / 解决方案同学的痛点

- 客户差异大，同一条分拣主流程经常因为组波、料口、打印、贴码、异常处理不同而出现大量变体。
- 过去更多依赖项目化开发承接差异，复用率低，产品边界难沉淀。
- 缺少统一的流程表达方式时，产品方案很难快速验证“这个客户到底改的是规则、流程，还是系统能力”。
- 面对客户临时变更，产品很难低成本评估影响范围和回归范围。

PS：

- 这不是单纯做一个拖图工具，而是在给产品建立“可复用的客户差异承接模型”。
- 平台化越晚做，后续每多一个客户，项目化成本都会继续线性上升。

### 5.2 测试同学的痛点

- 当前链路天然跨系统、跨角色、跨时序，包含同步调用、异步回调、人工确认和子流程嵌套，回归复杂度高。
- 一旦流程卡在 `waiting`，如果没有标准化的恢复事件和定位手段，测试验证成本会很高。
- 不同节点的输入、输出、状态补丁和恢复条件分散时，很难稳定复现问题。
- 现场问题如果只能靠日志拼接，测试很难形成标准化回归用例。

PS：
- 这个项目越往后越需要测试参与定义“场景矩阵”和“恢复事件回放”能力，不是后期补几个接口测试就够。
- 如果测试侧没有尽早参与异步场景建模，后续工业现场的问题会集中暴露。
- 调用外部时，在`header`里统一传 `run-id / request-id / workflow-id`，是测试和现场排障的基础设施需求。

### 5.3 交付 / 实施同学的痛点

- 现场联调对象多，涉及上游业务系统、设备系统、人工工位和回调链路。
- 客户差异很多时候不是代码问题，而是节点配置、上下文映射、回调事件和流程分支组合问题。
- 现场异常经常不是“系统挂了”，而是“流程卡住了、回调没回、状态不一致、人工环节没闭环”。
- 如果没有统一 runId、timeline、快照和补录机制，现场排障成本会很高。

PS：

- 平台价值的一半来自“交付可落地”，另一半来自“出问题时能快速查明白”。
- 现场排障成本下降，才是这个平台真正能争取实施同学支持的关键。

### 5.4 运维 / 支持同学的痛点

- 工业现场更关注稳定性和可恢复，而不仅是接口是否返回成功。
- 如果没有运行快照、版本绑定、终止能力、错误码和节点级定位，线上问题会快速演化为人工救火。
- 后续如果引入更多动态节点和子流程，缺少治理机制会直接增加生产风险。

PS：

- 平台不是只追求“能跑起来”，而是要追求“出了问题能恢复、能审计、能定位、能止损”。

### 5.5 真实案例(待补充)

补充以下素材：

- 产品侧案例：某客户因为组波或异常流转差异，导致项目反复改代码。
- 测试侧案例：某条等待回调或人工确认链路难以稳定回归。
- 交付侧案例：现场因为接口回调、状态不同步或排障困难导致上线延期。

## 6. 主要节点功能，以及为什么当前没有 While / Database

### 6.1 为什么这些节点是第一优先级

当前开放或已实现的节点，基本覆盖了分拣作业编排最核心的几类能力：

| 节点类别 | 代表节点 | 价值 |
| --- | --- | --- |
| 流程控制 | `Start`、`End`、`Switch`、`If/Else`、`Sub-Workflow` | 把复杂业务主链路拆成可组合流程骨架 |
| 数据获取 / 整理 | `Fetch`、`SetState` | 获取上下游数据，并把中间状态沉淀给后续节点 |
| 扩展执行 | `Shell`、`Code` | 用受控扩展方式承接短期定制需求 |
| 异步等待 | `Wait` | 承接设备回调、人工确认、第三方异步结果返回 |
| 任务协同 | `Task` | 把人工、PDA、外部系统任务统一抽象成可等待节点 |
| 触发响应 | `Webhook Trigger`、`Respond` | 给对外接入和回执输出预留统一协议入口 |

这批节点的共同特点是：

- 足以覆盖当前仓储分拣主链路。
- 能体现平台的差异化能力。
- 复杂度仍可控，适合当前阶段收敛节点协议和运行模型。

### 6.2 为什么当前不优先开放 `While`

`While` 不是不能做，而是不适合在当前阶段作为业务默认节点开放，原因主要有四点：

- 循环节点天然容易引入死循环、长时间占用和不可预估执行成本。
- 一旦和外部副作用混用，幂等、重试、终止和恢复都会变复杂。
- 工业现场更需要“可恢复、可解释、可定位”的事件驱动模型，而不是开放式循环脚本模型。
- 当前很多循环诉求，其实可以通过“子流程 + 外部事件重复触发 + 有界批处理”来替代。


### 6.3 为什么当前不开放通用 `Database` 节点

当前不建议把数据库直接暴露成业务可拖拽节点，原因如下：

- 会把流程和底层业务表结构直接耦合，破坏平台抽象层。
- 不利于多租户隔离、权限控制和审计治理。
- 出问题后很难区分是流程编排问题、数据问题还是环境问题。
- 平台应该鼓励“命名业务动作”或“标准连接器”，而不是鼓励业务流程直接写库。
- 对于确实需要数据访问的场景，优先通过 `Fetch`、连接器、业务服务 API 或受控扩展点承接。

### 6.4 节点开放策略建议

- 默认开放：`Start`、`End`、`Fetch`、`Switch`、`Wait`、`Task`、`Sub-Workflow`
- 高级用户开放：`Respond`、`SetState`
- 严格受控开放：`Code`、`Shell`
- 暂不开放：`While`、`Database`

这样既能保证平台可用，也能避免平台过早滑向脚本托管或底层能力直通。

## 7. 当前项目架构

### 7.1 当前实际架构

当前仓库的真实实现架构可以概括为：

`Vue 3 前端 -> Rust Backend(Axum) -> Rust Runner(库内嵌) -> PostgreSQL`

其中：

- 前端负责工作流编辑、运行调试、AI 会话展示。
- backend 负责统一暴露 `/runner-api` 和 `/views`。
- runner 不是独立服务，而是作为 Rust 库直接运行在 backend 进程内（考虑封装独立cli）。
- PostgreSQL 当前同时承接 workflow catalog、run summary、run snapshot、edit session 持久化。

### 7.2 当前分层说明

### 前端层

- 工作流列表页：查看流程和运行入口。
- 工作流编辑器：拖拽节点、配置参数、发布与导出。
- Run 模式：发起运行、查看状态和 timeline。
- AI 模式：展示 AI 编辑会话、会话状态和最新草稿预览。

### API / Backend 层

- 提供 workflow 注册、查询、事件订阅、run 启动、resume、terminate 等接口。
- 提供 SSE 事件流，用于工作流、run、edit session 的状态推送。
- 统一处理 CORS、静态资源托管和请求日志。

### Runner 内核层

- 采用“按节点类型分发的状态机”模型执行工作流。
- 通过 `ExecutorRegistry` 注册不同节点执行器。
- 节点执行和流程跳转分离，运行时统一维护 `input / state / timeline / signal`。
- `wait`、`task`、`sub_workflow` 等节点可进入等待态，并依赖 snapshot 恢复。

### 存储层

- `workflow_runs`：保存运行摘要、状态、当前节点、timeline。
- `workflow_snapshots`：保存等待恢复所需的触发数据、输入、状态、timeline 和 env。
- catalog store：保存 workflow 定义、workspace 信息。
- edit session store：保存 AI 编辑会话草稿和同步状态。

### 7.3 当前架构的优点

- 实现链路短，适合当前阶段快速迭代。
- Runner 直接以内嵌库方式接入 backend，调试和集成成本低。
- 已经具备流程定义、执行、等待、恢复、终止、时间线的最小闭环。
- 后续可以围绕任务中心、规则中心、连接器中心继续演进，而不需要推倒重来。

### 7.4 当前架构的边界

- 目前 runner 与 backend 同进程，隔离性和扩展性还有限。
- PostgreSQL 当前承担的职责较多，后续高并发和异步解耦能力仍需增强。
- 真正的任务中心、规则中心、连接器中心还没有完全独立成清晰子系统。
- 当前 AI 模式仍偏“会话协作展示”，还没有形成页面内原生 AI 交互体验。

## 8. 项目的核心难点

### 8.1 难点一：流程中的人工介入如何实现

人工介入不是简单插入一个“人工节点”就结束，真正的难点在于把人工操作纳入统一运行时模型。

需要同时解决的问题包括：

- 如何把人工动作抽象成统一任务，而不是散落在不同页面或工位系统里。
- 如何让流程在人工节点进入 `waiting`，又能在收到人工回执后恢复执行。
- 如何支持人工重试、跳过、转人工、补录回执、强制终止等操作。
- 如何记录操作人、操作时间、上下文和原因，满足审计要求。
- 如何把人工工位、PDA、外部作业终端统一纳入同一任务生命周期。

当前项目已经有较好的技术起点：

- `task` 节点已经能把节点转成等待态。
- `resume` 机制已经具备，可以承接人工完成事件。
- `timeline + snapshot` 已经为人工介入后的追踪和恢复打下基础。

但下一步仍需补齐：

- 独立任务模型与任务状态机。
- 人工介入动作集的产品定义。
- 工位 / PDA / 管理后台的统一回执协议。

### 8.2 难点二：如何保证工业场景下的可靠性

工业现场最怕的不是功能少，而是“跑着跑着卡住、出了问题查不清、人工接不了、恢复不了”。

要保证可靠性，至少需要同时守住以下几件事：

- 流程实例固定版本绑定，避免发布后影响在途实例。
- 可等待、可恢复，而不是只能顺序跑到底。
- 副作用节点具备幂等、超时、重试和终止能力。
- 运行中断后可基于 snapshot 恢复，而不是从头重跑。
- 所有关键节点都能留下 timeline、错误信息和上下文。
- 能按 `runId / workflowId / orderNo / requestId / taskId` 快速定位问题。

当前项目已经实现或具备雏形的能力包括：

- run summary 持久化
- snapshot 持久化
- waiting / resume
- terminate
- timeline
- 子流程执行

但要真正适配工业场景，还需要继续增强：

- 更细粒度的幂等策略
- 更完善的错误码和告警
- 并发任务控制与背压能力
- 现场排障工具化
- 更强的连接器隔离与超时治理

### 8.3 补充难点

- 多版本与在途实例兼容问题
- 子流程上下文传递和命名空间治理问题
- 扩展节点能力与平台治理之间的平衡问题
- 现场问题复现、回放和测试标准化问题

## 9. 后续计划


### 9.1 动态业务节点的添加方式

目标：

- 让新增业务能力不必每次都硬编码进主流程编辑器。

方向：

- 后端保持统一节点协议和执行器注册机制。
- 前端节点面板改成“元数据驱动”，通过 schema 描述表单、校验、分类和帮助信息。
- 节点能力按版本管理，支持灰度开放和权限控制。

价值：

- 降低新增业务节点的研发成本。
- 让“平台能力扩展”从改主代码逐步转向扩展点治理。

### 9.2 工作流并发与任务控制

目标：

- 支持工作流级、节点级的并发控制，并可按场景配置。

方向：

- 增加 workflow 级并发上限、队列等待、超量拒绝和背压策略。
- 对 `task / fetch / shell / code` 等高成本节点增加并发阈值控制。
- 为现场任务高峰提供更可控的资源调度能力。

价值：

- 降低高峰期资源打满和流程雪崩风险。
- 让平台从“能执行”提升到“能稳定承载”。

### 9.3 主 / 子流程上下文获取

目标：

- 让子流程既能拿到必要上下文，又不污染父流程状态。

方向：

- 明确 `trigger / input / state / env / parentRunId / workflowKey` 的作用域。
- 为主子流程建立命名空间化 state 管理和输入输出映射规范。
- 提供更直观的主子流程调试视图。

价值：

- 降低复杂流程嵌套后的理解成本和维护成本。
- 避免子流程接入后状态错乱或边界失控。

### 9.4 降低现场排障成本

目标：

- 把“靠人翻日志排障”逐步变成“靠工具定位问题”。

方向：

- 支持按订单号、波次号、runId、requestId、taskId 联查。
- timeline 中补充节点耗时、输入输出摘要、错误码和恢复建议。
- 支持恢复事件回放、人工补录和常见故障排查模板。
- 逐步建设针对实施和测试的排障工作台。

价值：

- 这是最容易争取交付、测试和运维支持的建设项。
- 排障成本下降后，平台才会真正被现场团队认可。

### 9.5 AI 会话框集成到页面

目标：

- 让 AI 协作从“页面展示会话信息”升级为“页面内直接发起协作和查看结果”。

方向：

- 在编辑器侧边栏或底部集成原生 AI 会话框。
- 将提示词、会话状态、变更摘要、预览同步集中到一个区域。
- 保留人工确认和回滚机制，避免 AI 直接覆盖正式版本。

价值：

- 降低 AI 协作门槛。
- 让业务人员和实施同学更容易把 AI 当成流程搭建助手，而不是外部工具。

### 9.6 补充的计划项

> 从业务场景出发，收集并补充更多后续计划项：

- 规则中心
- 自定义节点开发工具
- 标准连接器 / 业务动作中心
- 流程版本
- 测试沙箱、回放和模拟器能力
- 统一任务工作台与人工介入操作台

## 10. 总结

> 当前项目已经不是简单的工作流画布，而是具备了可运行、可等待、可恢复、可追踪的流程执行雏形。它已经证明这条路线能承接仓储/物流分拣场景的核心主链路，下一阶段的重点不应该只是继续堆节点，而应该围绕“人工介入、可靠性、排障效率、扩展治理”继续投入，把它从可用的 MVP 推进到可落地的工业级平台。
