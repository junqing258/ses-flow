# 登录认证与数据库表说明

本文档说明 SES Flow 当前后端的登录认证机制，以及启动时由代码自动初始化的 PostgreSQL 表结构。内容基于以下实现整理：

- `apps/backend/src/main.rs`
- `apps/backend/src/modules/auth.rs`
- `apps/backend/src/modules/routes.rs`
- `apps/backend/src/modules/system/system_store.rs`
- `apps/runner/src/store/postgres.rs`
- `apps/runner/src/store/catalog.rs`
- `apps/runner/src/store/session.rs`

## 1. 登录认证

### 1.1 启动与初始化

后端启动时会读取 `DATABASE_URL`，初始化 runner、catalog、edit session、system settings 与 auth 相关的 PostgreSQL store。认证默认启用，`ApiState.auth_required` 在正式入口中固定为 `true`。

启动阶段还会执行 `AuthService::bootstrap_from_env()`，用于创建或修复初始超级管理员：

| 环境变量 | 说明 |
| --- | --- |
| `SES_AUTH_BOOTSTRAP_USERNAME` | 初始超级管理员用户名。为空时跳过 bootstrap |
| `SES_AUTH_BOOTSTRAP_PASSWORD` | 初始超级管理员密码。为空时跳过 bootstrap |
| `SES_AUTH_BOOTSTRAP_EMAIL` | 初始超级管理员邮箱，可选 |

如果 bootstrap 用户已存在，系统会确保该用户拥有 `SUPER_ADMIN` 角色；如果不存在，则创建用户并分配 `SUPER_ADMIN`。

### 1.2 密码与 Token

- 密码使用 Argon2 哈希后写入 `auth_users.password_hash`。
- 登录成功后生成格式为 `ses_<uuid>_<uuid>` 的 access token。
- 数据库不保存明文 token，只保存 `SHA-256` 后的十六进制摘要 `auth_sessions.token_hash`。
- 会话有效期为 24 小时，由 `SESSION_TTL_HOURS` 控制。
- 每次认证成功会更新 `auth_sessions.last_used_at`。
- 登出会写入 `auth_sessions.revoked_at`，不会物理删除会话记录。

### 1.3 前端登录态

前端 `useAuth()` 调用 `/api/auth/login` 登录，并把 access token 持久化到浏览器存储中，键名为：

```text
ses.auth.access-token
```

应用初始化时，如果本地存在 token，会请求 `/api/auth/me` 恢复当前用户和会话；如果返回 401，则清空本地登录态。

### 1.4 Auth API

认证相关接口挂载在 `/api/auth`：

| 方法 | 路径 | 说明 | 权限 |
| --- | --- | --- | --- |
| `POST` | `/login` | 普通账号登录 | 公开 |
| `POST` | `/register` | 公共注册入口 | 当前禁用，始终返回 403 |
| `GET` | `/me` | 获取当前用户与会话 | 需要有效 Bearer token |
| `POST` | `/logout` | 登出并撤销当前会话 | 需要有效 Bearer token |
| `POST` | `/station-login` | 工作站登录，会把会话绑定到 station/platform | 账号需有 `workstation.login` 且有站点授权 |
| `POST` | `/station-authorize` | 校验当前工作站会话是否有指定权限与站点授权 | 默认要求 `workstation.operate` |
| `POST` | `/admin/users` | 创建用户 | `auth.manage_users` |
| `POST` | `/admin/users/{user_id}/active` | 启用或停用用户 | `auth.manage_users` |
| `POST` | `/admin/users/{user_id}/roles` | 分配用户角色 | `auth.manage_users` |
| `POST` | `/admin/users/{user_id}/station-grants` | 分配工作站授权 | `auth.manage_users` |

新创建的用户会自动获得一条默认工作站授权：

- `station_id='*'`
- `platform_id=NULL`
- `grant_type='workstation'`

这表示新用户默认可匹配所有工作站；如果后续需要收紧范围，可以再通过管理接口补充更细粒度的授权策略，或直接调整库中授权数据。

请求认证使用标准 Bearer 头：

```http
Authorization: Bearer <access_token>
```

`EventSource` 不能设置请求头，因此 `/runner-api` 的 SSE 类请求也支持通过查询参数传 token：

```text
?token=<access_token>
```

### 1.5 Runner API 权限规则

`/runner-api/health` 公开；`/runner-api/runs/{run_id}/resume` 也作为外部回调/恢复入口公开。其他 `/runner-api` 请求会经过权限中间件。

| 路径规则 | 权限 |
| --- | --- |
| `/system/*`、`/plugin-registrations`、`/catalog/refresh` | `plugin.manage` |
| `GET /edit-sessions*` | `workflow.read` |
| `POST/PUT/PATCH /edit-sessions*` | `workflow.write` |
| `/workflows/*/run` | `workflow.run` |
| `GET /workflows*` | `workflow.read` |
| `POST/PUT/PATCH /workflows*` | `workflow.write` |
| `/runs/*/terminate`、`/runs/*/manual-patch` | `workflow.operate` |
| `/runs*` | `workflow.read` |
| `/node-descriptors*` | `workflow.read` |

| `workflow_definitions` | runner/catalog | 工作流定义与前端画布文档 |
| `workflow_edit_sessions` | runner/edit session | AI 编辑会话草稿 |
| `workflow_runs` | runner/run | 工作流运行摘要 |
| `workflow_snapshots` | runner/run | 可恢复运行快照 |
| `chute_status` | runner/sorting | 分拣格口状态 |

| `sorting_flow` | runner/sorting | 分拣流水 |

## 3. 认证相关表

### 3.1 `auth_users`

用户主表。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `TEXT PRIMARY KEY` | 用户 ID，UUID 字符串 |
| `username` | `TEXT NOT NULL` | 用户名 |
| `email` | `TEXT` | 邮箱，可为空 |
| `password_hash` | `TEXT NOT NULL` | Argon2 密码哈希 |
| `display_name` | `TEXT` | 展示名 |
| `is_active` | `BOOLEAN NOT NULL DEFAULT TRUE` | 是否启用 |
| `password_changed_at` | `TIMESTAMPTZ NOT NULL DEFAULT NOW()` | 密码变更时间，当前预留 |
| `last_login_at` | `TIMESTAMPTZ` | 最近登录时间 |
| `created_at` | `TIMESTAMPTZ NOT NULL DEFAULT NOW()` | 创建时间 |
| `updated_at` | `TIMESTAMPTZ NOT NULL DEFAULT NOW()` | 更新时间 |

索引：

- `idx_auth_users_username_lower`：`lower(username)` 唯一索引。
- `idx_auth_users_email_lower`：`lower(email)` 条件唯一索引，邮箱非空时生效。

### 3.2 `auth_sessions`

登录会话表。普通登录和工作站登录都会写入该表。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `TEXT PRIMARY KEY` | 会话 ID |
| `user_id` | `TEXT NOT NULL` | 关联 `auth_users.id`，用户删除时级联删除 |
| `token_hash` | `TEXT NOT NULL UNIQUE` | access token 的 SHA-256 摘要 |
| `station_id` | `TEXT` | 工作站会话绑定的站点 ID |
| `platform_id` | `TEXT` | 工作站会话绑定的平台 ID |
| `grant_type` | `TEXT` | 授权类型，工作站登录默认为 `workstation` |
| `user_agent` | `TEXT` | 登录请求 User-Agent |
| `ip_address` | `TEXT` | 登录请求 `x-forwarded-for` |
| `expires_at` | `TIMESTAMPTZ NOT NULL` | 会话过期时间 |
| `revoked_at` | `TIMESTAMPTZ` | 登出或撤销时间 |
| `last_used_at` | `TIMESTAMPTZ` | 最近一次认证成功时间 |
| `created_at` | `TIMESTAMPTZ NOT NULL DEFAULT NOW()` | 创建时间 |

索引：

- `idx_auth_sessions_user_id`
- `idx_auth_sessions_expires_at`

### 3.3 `auth_roles`

| `updated_at` | `TIMESTAMPTZ NOT NULL DEFAULT NOW()` | 更新时间 |

### 3.5 `auth_user_roles`

| 字段 | 类型 | 说明 |

### 3.7 `auth_station_grants`

工作站授权表，用于限制用户可登录或操作哪些 station/platform。`station_id = '*'` 表示通配所有工作站；`platform_id IS NULL` 表示不限制平台。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `TEXT PRIMARY KEY` | 授权 ID |
| `user_id` | `TEXT NOT NULL` | 关联 `auth_users.id`，用户删除时级联删除 |
| `station_id` | `TEXT NOT NULL` | 工作站 ID，支持 `*` 通配 |
| `platform_id` | `TEXT` | 平台 ID，可为空 |
| `grant_type` | `TEXT NOT NULL` | 授权类型，工作站登录默认为 `workstation` |
| `is_active` | `BOOLEAN NOT NULL DEFAULT TRUE` | 授权是否启用 |
| `created_at` | `TIMESTAMPTZ NOT NULL DEFAULT NOW()` | 创建时间 |
| `updated_at` | `TIMESTAMPTZ NOT NULL DEFAULT NOW()` | 更新时间 |

约束与索引：

- 唯一约束：`(user_id, station_id, platform_id, grant_type)`。
- `idx_auth_station_grants_lookup`：用于登录和授权校验查询。

默认写入规则：

- 管理员通过 `/api/auth/admin/users` 创建用户后，系统会自动写入一条 `('*', NULL, 'workstation')` 授权。
- 通过环境变量 bootstrap 首次创建超级管理员时，也会自动写入同样的默认授权。

## 8. 关系与排障建议

### 8.1 认证关系

- `auth_users` 通过 `auth_user_roles` 关联 `auth_roles`。
- `auth_roles` 通过 `auth_role_permissions` 关联 `auth_permissions`。
- `auth_sessions.user_id` 指向 `auth_users.id`。
- `auth_station_grants.user_id` 指向 `auth_users.id`。

登录失败排查顺序：

1. 检查用户是否存在：`auth_users.username` 或 `auth_users.email`。
2. 检查用户是否启用：`auth_users.is_active = true`。
3. 检查用户是否拥有角色：`auth_user_roles`。
4. 检查角色是否拥有所需权限：`auth_role_permissions` 与 `auth_permissions.code`。
5. 工作站登录额外检查 `auth_station_grants` 是否存在匹配的 `station_id/platform_id/grant_type`。
6. 已登录请求检查 `auth_sessions.expires_at`、`revoked_at` 与 token 摘要是否匹配。

### 8.2 工作流关系

- `workspaces` 是 `workflow_definitions` 的上级。
- `workflow_definitions.definition` 是 runner 执行定义；`editor_document` 是前端画布文档。
- `workflow_edit_sessions` 保存 AI 编辑过程中的草稿，不直接覆盖 `workflow_definitions`。
- `workflow_runs` 保存运行摘要；`workflow_snapshots` 保存可恢复运行上下文。

### 8.3 分拣关系

- `wave` 表示一个波次及其绑定格口。
- `wave_order` 记录波次内订单与运行实例。
- `sorting_flow` 记录更细的分拣流水。
- `chute_status` 保存格口当前状态和容量信息。
