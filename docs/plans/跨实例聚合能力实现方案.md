# 跨实例聚合能力实现方案

## 1. 问题背景

当前 runner 的执行模型是**单实例封闭**：每次 `POST /workflows/{id}/run` 启动一个独立的 `WorkflowRunRecord`，run 的 `state` 仅属于该实例，不对外共享。

仓储分拣场景中，多个 run 实例需要协同操作同一份物理资源：

| 场景 | 需要共享的数据 | 并发问题 |
|---|---|---|
| 格口件数累加 | `chute_status.current_count` | 多个分拣 run 同时写同一格口 |
| 格口满触发集包 | `chute_status.is_full` | 防止多 run 重复触发同一格口集包 |
| 波次聚合订单 | `wave.order_list` | 多订单并发关联同一波次 |
| 小车资源状态 | 小车当前归属的 run / platformId | 防止同一小车被多个 run 同时派遣 |

---

## 2. 技术选型

### 现有栈

- **PostgreSQL**：已有 `sqlx` + `PgPool`，`PostgresRunStore` / `PostgresCatalogStore` 均已在用
- **Tokio**：异步运行时，内置 `tokio::sync` 并发原语
- **无 Redis**：docker-compose 中未引入，引入有额外运维成本

### 选型结论

**使用 PostgreSQL 作为跨实例共享状态的唯一 source of truth**，辅以进程内 `tokio::sync::RwLock` 读缓存应对热路径查询。

| 维度 | PG | Redis | 内存 |
|---|---|---|---|
| 新增依赖 | 无（已有） | 新增 crate + 服务 | 无 |
| 持久化 | ✅ | ✅（需配置） | ❌ 重启丢失 |
| 并发安全 | ✅ `FOR UPDATE` / CAS | ✅ Lua 脚本 | ⚠️ 单进程可用 |
| 多副本支持 | ✅ | ✅ | ❌ |
| 查询能力 | ✅ SQL 全量 | ⚠️ 有限 | ⚠️ 有限 |
| 当前阶段适配 | ✅ 最优 | ❌ 过度引入 | ❌ 不可持久 |

Redis 留到**多副本横向扩展阶段**再引入，届时将内存读缓存层替换为 Redis，PG 仍作为 source of truth，改动范围最小。

## 3. 数据模型

在 `PostgresRunStore.init_schema()` 中新增四张表，与现有 `workflow_runs` 同库同连接池。

### 3.1 格口状态表 `chute_status`

```sql
CREATE TABLE IF NOT EXISTS chute_status (
    chute_id        TEXT        PRIMARY KEY,
    platform_id     TEXT        NOT NULL,
    wave_no         TEXT,
    current_count   INTEGER     NOT NULL DEFAULT 0,
    capacity        INTEGER     NOT NULL DEFAULT 100,
    is_full         BOOLEAN     NOT NULL DEFAULT FALSE,
    pack_run_id     TEXT,                          -- 正在执行集包的 run_id，防重复触发
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX IF NOT EXISTS idx_chute_status_platform_id ON chute_status (platform_id);
```

### 3.2 波次表 `wave`

```sql
CREATE TABLE IF NOT EXISTS wave (
    wave_no         TEXT        PRIMARY KEY,
    platform_id     TEXT        NOT NULL,
    chute_id        TEXT        NOT NULL,
    rule_version    TEXT,
    status          TEXT        NOT NULL DEFAULT 'active',  -- active | packing | completed
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

### 3.3 波次订单关联表 `wave_order`

```sql
CREATE TABLE IF NOT EXISTS wave_order (
    wave_no         TEXT        NOT NULL,
    run_id          TEXT        NOT NULL,
    order_id        TEXT        NOT NULL,
    status          TEXT        NOT NULL DEFAULT 'sorting', -- sorting | sorted | packed
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    PRIMARY KEY (wave_no, run_id)
);

CREATE INDEX IF NOT EXISTS idx_wave_order_wave_no ON wave_order (wave_no);
CREATE INDEX IF NOT EXISTS idx_wave_order_run_id  ON wave_order (run_id);
```

### 3.4 分拣流水表 `sorting_flow`

每次分拣动作的审计日志，仅追加写入，不更新。

```sql
CREATE TABLE IF NOT EXISTS sorting_flow (
    id              BIGSERIAL   PRIMARY KEY,
    task_id         TEXT        NOT NULL,              -- 本次分拣任务唯一标识（由 run 生成）
    run_id          TEXT        NOT NULL,
    platform_id     TEXT        NOT NULL,
    wave_no         TEXT,
    order_id        TEXT,
    chute_id        TEXT        NOT NULL,
    sku             TEXT,
    barcode         TEXT,
    sorting_result  TEXT        NOT NULL DEFAULT 'ok', -- ok | error | missing
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX IF NOT EXISTS idx_sorting_flow_run_id     ON sorting_flow (run_id);
CREATE INDEX IF NOT EXISTS idx_sorting_flow_wave_no    ON sorting_flow (wave_no);
CREATE INDEX IF NOT EXISTS idx_sorting_flow_platform_id ON sorting_flow (platform_id);
CREATE INDEX IF NOT EXISTS idx_sorting_flow_created_at ON sorting_flow (created_at);
```

## 4. 核心操作与并发安全

### 4.1 格口件数累加（原子 CAS）

分拣完成后，run 通过 `fetch` 节点调用内部 API，由 backend 执行：

```sql
UPDATE chute_status
SET
    current_count = current_count + 1,
    is_full       = (current_count + 1 >= capacity),
    updated_at    = NOW()
WHERE chute_id = $1
RETURNING chute_id, current_count, capacity, is_full;
```

- 单条 `UPDATE ... RETURNING` 是原子操作，无需显式事务
- 返回 `is_full` 供调用方决定是否触发集包

### 4.2 集包抢占（防止多 run 重复触发）

多个分拣 run 同时检测到格口满时，只允许一个 run 触发集包：

```sql
UPDATE chute_status
SET pack_run_id = $1, updated_at = NOW()
WHERE chute_id = $2
  AND is_full = TRUE
  AND pack_run_id IS NULL          -- 尚未被其他 run 抢占
RETURNING chute_id;
```

- 返回行数为 1：当前 run 抢到，启动 `plugin:pack_task`
- 返回行数为 0：已有其他 run 在处理，当前 run 直接结束

### 4.3 波次订单关联（幂等写入）

```sql
INSERT INTO wave_order (wave_no, run_id, order_id)
VALUES ($1, $2, $3)
ON CONFLICT (wave_no, run_id) DO NOTHING;
```

### 4.4 波次状态推进（乐观锁）

```sql
UPDATE wave
SET status = 'packing', updated_at = NOW()
WHERE wave_no = $1
  AND status  = 'active'           -- 只有 active 状态才能推进
RETURNING wave_no;
```

## 5. 服务层设计

在 `apps/backend/src/modules/` 下新增 `shared_state` 模块，对外暴露内部 HTTP API（仅 runner 调用，不对外）：

```
POST   /internal/chutes/{chute_id}/increment          格口件数+1，返回 is_full
POST   /internal/chutes/{chute_id}/claim-pack         抢占集包权，返回是否成功
GET    /internal/chutes/{chute_id}                    查询格口状态
POST   /internal/waves                                创建波次
GET    /internal/waves/{wave_no}                      查询波次
POST   /internal/waves/{wave_no}/orders               关联订单到波次
POST   /internal/sorting-flows                        写入分拣流水
```

workflow 通过 `fetch` 节点调用以上接口，runner 本身不直接访问 PG 业务表，保持 runner 与业务逻辑解耦。

```
fetch节点 → POST /internal/chutes/{id}/increment
         ← { chute_id, current_count, capacity, is_full }

if_else节点 → 判断 is_full
  true  → fetch节点 → POST /internal/chutes/{id}/claim-pack
               ← { claimed: true }  → plugin:pack_task
               ← { claimed: false } → end（其他 run 已处理）
  false → end
```

## 6. 读缓存（热路径优化）

小车到达查询（每次扫码后都需查询）属于读多写少场景，在 backend 进程内用 `tokio::sync::RwLock<HashMap>` 缓存格口状态：

```rust
pub struct ChuteStatusCache {
    inner: RwLock<HashMap<String, ChuteStatusEntry>>,
    ttl: Duration,
}
```

- **写穿策略**：`increment` / `claim-pack` 写 PG 后同步更新缓存
- **TTL 失效**：entry 超过 TTL（建议 30s）时下次读触发 PG 回源
- **缓存粒度**：按 `chute_id` 独立缓存，互不干扰

多副本阶段：将 `ChuteStatusCache` 替换为 Redis Hash，接口不变，只换底层实现。

## 7. 与现有 runner 的集成点

| 集成位置 | 方式 | 说明 |
|---|---|---|
| 格口件数累加 | `fetch` 节点 → `/internal/chutes/{id}/increment` | RCS 分拣完成后触发 |
| 集包抢占 | `fetch` 节点 → `/internal/chutes/{id}/claim-pack` | 格口满后竞争触发 |
| 波次创建 | `fetch` 节点 → `/internal/waves` | 组波完成时写入 |
| 订单关联波次 | `fetch` 节点 → `/internal/waves/{no}/orders` | 扫码完成后写入 |
| 格口状态查询 | `fetch` 节点 → `GET /internal/chutes/{id}` | 任意节点按需查询 |
| 分拣流水写入 | `fetch` 节点 → `POST /internal/sorting-flows` | 每次分拣动作完成后追加 |

runner 侧无需改动，全部通过已有的 `fetch` 节点 + HTTP 调用实现。

## 8. 实施顺序

1. **在 `init_schema()` 中追加建表语句**（`chute_status` / `wave` / `wave_order` / `sorting_flow`）
2. **实现 `SharedStateStore` trait**，封装 PG 操作（increment / claim-pack / wave CRUD）
3. **新增 `shared_state` 模块**，注册内部路由（`/internal/*`，不走公开鉴权中间件）
4. **添加 `ChuteStatusCache`**（`RwLock<HashMap>`）并挂载到 Axum `AppState`
5. **集成测试**：用两个并发 `fetch` 请求验证 increment 原子性和 claim-pack 互斥性
6. **在仓储场景 workflow 中接入**，替换 `fetch` 节点 URL

## 9. 未来演进

```
当前（单副本）                    多副本阶段
─────────────────────────────    ─────────────────────────────
PG source of truth           →   PG source of truth（不变）
tokio RwLock 读缓存          →   Redis Hash 读缓存
无消息总线                   →   Redis Stream 事件总线（可选）
```

PG 层接口与数据模型保持稳定，多副本改造只涉及缓存层替换，业务逻辑和 workflow 定义均无需变更。
