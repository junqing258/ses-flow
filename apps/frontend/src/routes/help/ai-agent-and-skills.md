# AI Agent 模式与 Skill 说明

本页说明 SES Flow 中的 AI Agent 模式是如何工作的，以及 `ses-flow-skill` 在整个协作链路中的角色。

## AI Agent 模式的定位

AI Agent 模式的目标不是在浏览器里提供一个普通聊天框，而是建立一条“当前工作流草稿 <-> 外部 Agent <-> Web 预览”的协作链路。

换句话说：

- Web 端负责展示当前会话状态和预览结果。
- 外部 Agent 负责读取草稿、修改工作流、再把更新推回会话。

## Web 端当前会展示什么

进入 AI 模式后，右侧面板当前会展示以下信息：

- AI 会话状态。
- 最近同步时间。
- `runner_base_url`
- `session_id`
- 一段可直接给 Agent 使用的提示词示例。

这些信息的作用是让外部 Agent 能定位到“当前这一个工作流编辑会话”。

## AI 模式的基本使用流程

1. 在工作流编辑器里切换到 `AI` 模式。
2. 等待系统创建或恢复当前工作流的 AI 编辑会话。
3. 复制面板中的 `runner_base_url` 和 `session_id`。
4. 将需求连同这两个参数一起交给外部 Agent。
5. 外部 Agent 读取当前草稿、完成修改、推送最新预览。
6. Web 端通过 SSE 或轮询拿到更新后的会话内容并展示。

## 什么是 `ses-flow-skill`

`ses-flow-skill` 可以理解为“面向 SES Flow 编辑会话的专用工作流 Skill”。

它的主要作用是帮助 Agent 按正确步骤处理当前会话，例如：

- 读取当前 `session_id` 对应的最新草稿。
- 基于用户需求修改 workflow 或 editorDocument。
- 把修改后的草稿推送回 Runner 的编辑会话。
- 让 Web 端能及时刷新只读预览。

## 为什么需要 Skill

如果没有专用 Skill，外部 Agent 很容易出现以下问题：

- 不知道应该读取哪一个会话。
- 直接修改了本地文件，但没有同步回当前编辑会话。
- 生成了新流程，却没有基于当前草稿增量修改。
- Web 端看不到最新结果，造成“Agent 说改了，但页面没更新”。

因此，对 AI 协作场景来说，Skill 不是可有可无的装饰，而是确保链路闭环的关键。

## 推荐给 Agent 的说明方式

当前界面里已经给出了一段提示词模板，实际使用时建议至少包含这些信息：

- 说明这是 SES Flow 工作流编辑任务。
- 指定必须使用 `ses-flow-skill`。
- 提供 `runner_base_url`。
- 提供 `session_id`。
- 说明需要先读取当前草稿，再基于需求修改。

## AI 模式适合处理什么需求

比较适合：

- 批量新增或调整节点。
- 改一组映射、分支或节点配置。
- 根据文字需求快速改出一个可预览的草稿版本。

不太适合：

- 只改一个非常细的小字段，而且你已经知道具体位置。
- 需要强依赖人工确认的复杂业务规则，没有明确输入说明。

## 当前边界与限制

为了避免误解，下面几点需要明确：

- AI 模式下的真正编辑动作不在浏览器内完成。
- Web 端当前更像“会话状态展示器 + 预览接收器”。
- 如果 Runner 不可达，AI 会话创建、读取和同步都会失败。
- 即使进入 AI 模式，也仍然建议保留 Edit / Run 模式做最终人工检查。
