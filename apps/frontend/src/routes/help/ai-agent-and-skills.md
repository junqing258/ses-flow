# AI 会话模式说明

本页说明 SES Flow 中的 AI 模式是如何工作的，以及页面内会话协作和草稿预览是如何串起来的。

## AI 会话模式的定位

当前版本的 AI 模式已经改成页面内会话。你不需要再把 `runner_base_url`、`session_id` 或提示词模板复制给外部 Agent，而是直接在编辑器右侧面板输入需求，让 Agent 基于当前工作流草稿继续修改。

现在的协作链路可以理解为：

- 当前工作流草稿作为上下文被送入 edit session。
- 页面内会话承接你的需求、Agent 的回复和工具状态。
- 更新后的工作流预览会自动同步回编辑器。

## Web 端当前会展示什么

进入 AI 模式后，右侧面板当前会展示以下信息：

- Agent 当前状态。
- 最近同步时间。
- 消息记录。
- 工具执行状态。
- `agent_session_id`（如果当前会话已经建立）。
- 错误信息或连接状态。

这些信息的作用是帮助你直接在页面里判断当前协作是否正常、草稿是否已经同步。

## AI 模式的基本使用流程

1. 在工作流编辑器里切换到 `AI` 模式。
2. 先在右上角设置里填写 AI 供应商配置，包括 `ANTHROPIC_BASE_URL`、`ANTHROPIC_AUTH_TOKEN`、`ANTHROPIC_MODEL`。
3. 等待系统创建或恢复当前工作流的 AI 编辑会话。
4. 直接在右侧输入框描述修改需求并发送。
5. Agent 基于当前草稿执行修改，并把中间状态和结果回写到会话。
6. Web 端通过 Agent 事件流或轮询拿到更新后的内容并刷新预览。

## 当前会话里哪些上下文是系统自动维护的

虽然底层仍然会维护 edit session，但这些上下文现在由系统自动处理，不需要手动传递给外部工具：

- 当前工作流对应的 edit session。
- 会话关联的预览草稿。
- Agent 事件流连接状态。
- 最近一次预览同步时间。

这也是当前帮助页和旧说明最大的区别：用户面对的是页面内会话，不再是“复制参数给外部 Agent”的工作流。

## 旧版外部 Agent 流程不再是主流程

历史上，AI 协作曾经更偏向“外部 Agent + 会话同步”的模式，因此会提到 `ses-flow-skill`、`runner_base_url`、`session_id` 和提示词模板。

在当前页面内会话版本中，这些步骤都不是用户主流程，帮助说明也不再要求你：

- 手动复制会话参数。
- 准备单独的提示词交接给外部 Agent。
- 自己决定如何把结果同步回当前页面。

## AI 模式适合处理什么需求

比较适合：

- 批量新增或调整节点。
- 改一组映射、分支或节点配置。
- 根据文字需求快速改出一个可预览的草稿版本。
- 在一轮修改后继续追问，例如“保留刚才的分支，再补一个失败兜底”。

不太适合：

- 只改一个非常细的小字段，而且你已经知道具体位置。
- 需要强依赖人工确认的复杂业务规则，没有明确输入说明。

## 当前边界与限制

为了避免误解，下面几点需要明确：

- AI 模式依赖右上角保存的 AI 供应商配置，配置不完整时无法发送消息。
- 如果 Runner 不可达，AI 会话创建、读取和同步都会失败。
- 如果 Agent 事件流中断，页面会退回重连或轮询刷新。
- AI 结果会直接影响当前草稿预览，所以仍然建议在 Edit / Run 模式做最终人工检查。

## 对当前帮助结构的补充建议

如果后续继续完善这一部分，建议增加两个附加专题：

- Prompt 示例库：按“新增节点、改分支、调试等待节点、补响应节点”等场景给出高质量需求描述范例。
- 会话故障排查：如 AI 配置缺失、事件流未连接、预览未刷新、Runner 服务不可达。
