# 系统功能说明

本页说明 SES Flow 当前的主要使用方式，重点覆盖编辑模式、运行模式和 AI Agent 模式。

## 整体定位

SES Flow 当前是一套面向工作流编排的可视化编辑与运行预览界面，前端主要承担三类职责：

- 创建和编辑工作流草稿。
- 将工作流同步到 Runner 并发起运行。
- 在 AI Agent 模式下展示会话信息、预览结果与同步状态。

## 主要入口

系统当前的主入口集中在以下位置：

- [工作流列表](/views/#/workflow-list)：查看已有 workflow、模板和运行入口。
- [新建工作流](/views/#/workflow/new)：创建新的流程草稿。
- `/views/#/workflow/:id`：进入某个工作流的编辑器页面。

## 编辑器的三种页面模式

工作流编辑器顶部提供三种模式切换：

| 模式 | 适用场景 | 当前能力 |
| --- | --- | --- |
| Edit | 手动拖拽和配置节点 | 使用左侧节点面板搭建流程，编辑节点参数与连线 |
| Run | 运行与查看结果 | 发起运行、查看执行结果和 timeline，适合调试 |
| AI | 通过外部 Agent 协作编辑 | 展示会话状态、`runner_base_url`、`session_id` 和同步结果 |

## Edit 模式

Edit 模式是最基础的工作方式，适合人工搭建流程。

### 当前可完成的操作

- 从左侧面板拖拽节点到画布。
- 配置节点基础参数、输入映射、重试策略等字段。
- 调整节点连线和分支。
- 将当前流程同步并发布到 Runner。
- 导出当前流程的 JSON 结构。

### 适合什么时候用

- 初次搭建工作流骨架。
- 校验节点关系是否符合预期。
- 对单个节点配置做精确调整。

## Run 模式

Run 模式用于验证流程是否能按预期执行。

### 当前重点能力

- 触发当前工作流运行。
- 查看运行状态，例如 `running`、`completed`、`waiting`、`failed`。
- 查看 timeline 中每个节点的执行信息。
- 从工作流列表或编辑器中打开已有运行记录。

### 使用建议

- 流程结构调整后，优先进入 Run 模式做一次回归验证。
- 如果运行停在 `waiting`，说明流程正在等待外部事件或任务回调。

## AI Agent 模式

AI Agent 模式不是浏览器内直接聊天编辑，而是“前端展示 + 外部 Agent 实际修改”的协作模式。

### Web 端当前承担的职责

- 创建或读取 AI 编辑会话。
- 显示会话同步状态。
- 展示 `runner_base_url` 与 `session_id`。
- 展示给外部 Agent 使用的提示词示例。
- 通过轮询与 SSE 获取最新草稿和预览结果。

### 适合什么时候用

- 你已经知道要改什么，希望通过 AI 更快地批量改节点、改连线、改配置。
- 你需要让外部 Agent 基于当前草稿继续迭代，而不是从头生成流程。

## 发布与 JSON 导出

编辑器右上角当前还提供两个重要动作：

- `JSON`：导出当前工作流结构，适合检查底层定义或做排查。
- `Publish`：将流程发布到 Runner，形成可运行版本。

这两个动作适合在“结构大体完成”后使用，而不是一开始就频繁依赖。

## 当前边界与限制

为了避免帮助说明和真实能力不一致，这里明确列出几个当前边界：

- Webhook 触发相关字段已经可以配置，但 Runner 当前仍通过统一的 `POST /runner-api/workflows/<workflow_id>/run` 入口启动执行。
- AI Agent 模式下，浏览器并不是直接执行编辑逻辑的主体，真正的修改动作来自外部 Agent。
- 有些配置项已经在 UI 中占位，但还需要和运行时进一步联动完善。
