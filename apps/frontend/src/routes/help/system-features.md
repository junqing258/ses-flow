# 系统功能说明

本页说明 SES Flow 当前的主要使用方式，重点覆盖编辑模式、运行模式和 AI 会话模式。

## 整体定位

SES Flow 当前是一套面向工作流编排的可视化编辑与运行预览界面，前端主要承担三类职责：

- 创建和编辑工作流草稿。
- 将工作流同步到 Runner 并发起运行。
- 在 AI 模式下承接页面内会话、展示协作消息、预览结果与同步状态。

## 主要入口

系统当前的主入口集中在以下位置：

- [工作流列表](/views/#/workflow-list)：查看已有 workflow、模板和运行入口。
- [新建工作流](/views/#/workflow/new)：创建新的流程草稿。
- `/views/#/workflow/:id`：进入某个工作流的编辑器页面。

## 编辑器的三种页面模式

工作流编辑器顶部提供三种模式切换：

| 模式 | 适用场景 | 当前能力 |
| --- | --- | --- |
| Edit | 手动拖拽和配置节点 | 使用左侧节点面板搭建流程，编辑节点参数与连线 |
| Run | 运行与查看结果 | 发起运行、查看执行结果和 timeline，适合调试 |
| AI | 通过页面内会话协作编辑 | 直接发送需求、查看协作消息、同步草稿预览 |

## Edit 模式

Edit 模式是最基础的工作方式，适合人工搭建流程。

### 当前可完成的操作

- 从左侧面板拖拽节点到画布。
- 配置节点基础参数、输入映射、重试策略等字段。
- 调整节点连线和分支。
- 将当前流程同步并发布到 Runner。
- 导出当前流程的 JSON 结构。

### 适合什么时候用

- 初次搭建工作流骨架。
- 校验节点关系是否符合预期。
- 对单个节点配置做精确调整。

## Run 模式

Run 模式用于验证流程是否能按预期执行。

### 当前重点能力

- 触发当前工作流运行。
- 查看运行状态，例如 `running`、`completed`、`waiting`、`failed`。
- 查看 timeline 中每个节点的执行信息。
- 从工作流列表或编辑器中打开已有运行记录。

### 使用建议

- 流程结构调整后，优先进入 Run 模式做一次回归验证。
- 如果运行停在 `waiting`，说明流程正在等待外部事件或任务回调。

## AI 会话模式

AI 模式现在就是浏览器内直接发起协作的工作方式。你在右侧面板输入需求后，系统会基于当前工作流的 edit session 调用 Agent，并把更新后的草稿预览同步回页面。

### Web 端当前承担的职责

- 创建或恢复当前工作流的 AI 编辑会话。
- 展示消息记录、工具状态和最近同步时间。
- 通过 Agent 事件流持续更新协作状态。
- 通过轮询与 SSE 获取最新草稿和预览结果。
- 将底层 `session_id` 等上下文保留在系统内部，不要求手动复制参数。

### 适合什么时候用

- 你已经知道要改什么，希望通过 AI 更快地批量改节点、改连线、改配置。
- 你希望直接在当前页面持续追问、补充约束，并基于当前草稿继续迭代。
- 你需要先拿到一个可预览版本，再回到 Edit / Run 模式做人工校对。

## 发布与 JSON 导出

编辑器右上角当前还提供两个重要动作：

- `JSON`：导出当前工作流结构，适合检查底层定义或做排查。
- `Publish`：将流程发布到 Runner，形成可运行版本。

这两个动作适合在“结构大体完成”后使用，而不是一开始就频繁依赖。

## 当前边界与限制

为了避免帮助说明和真实能力不一致，这里明确列出几个当前边界：

- Webhook 触发相关字段已经可以配置，但 Runner 当前仍通过统一的 `POST /runner-api/workflows/<workflow_id>/run` 入口启动执行。
- AI 模式依赖右上角设置中的 AI 供应商配置；如果 `base URL`、`auth token` 或 `model` 不完整，将无法发送消息。
- AI 返回的草稿会直接更新当前预览，但仍建议在 Edit / Run 模式做最终人工检查。
- 如果 Runner 或 Agent 事件流不可达，AI 会话的创建、同步和预览刷新都会失败。
- 有些配置项已经在 UI 中占位，但还需要和运行时进一步联动完善。
